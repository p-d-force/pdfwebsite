# Database Setup for Parent Data Force

This document describes how to set up MariaDB/MySQL for the Parent Data Force admin system, both locally for development and on production hosting (cPanel).

## Local Development with Docker

The easiest way to run the database locally is using Docker Compose.

### Prerequisites

- [Docker Desktop](https://www.docker.com/products/docker-desktop/) (Windows/Mac) or Docker Engine (Linux)
- Python 3.11+ (for running test scripts)
- Git (optional)

### Quick Start

1. **Start the database stack**:
   ```bash
   # Windows
   scripts\start-database.bat

   # Linux/macOS/Git Bash
   chmod +x scripts/start-database.sh
   scripts/start-database.sh
   ```

2. **Access the services**:
   - **phpMyAdmin**: http://localhost:8080 (database management)
   - **PHP Admin**: http://localhost:8081/admin/login.php (admin interface)
   - **MariaDB**: localhost:3306 (direct database connection)

3. **Login credentials**:
   - Default admin user: `admin` / `admin`
   - **IMPORTANT**: Change the password after first login!

### What gets set up

The Docker Compose stack includes:

1. **MariaDB 10.11** (matching production hosting version)
   - Database: `pdf_db` (configurable in `.env`)
   - User: `pdf_user` (configurable in `.env`)
   - Auto-imports:
     - Core schema (`backend/schema.sql`)
     - Admin tables (`backend/admin_schema.sql`)
     - Seed data from existing cases (`backend/seed_from_metadata.sql`)

2. **PHP 8.3 with Apache**
   - Serves the `admin/` directory at http://localhost:8081
   - PDO MySQL extension enabled
   - Apache rewrite module enabled
   - Custom PHP configuration

3. **phpMyAdmin**
   - Web-based database management
   - Pre-configured to connect to MariaDB

### Manual Docker Commands

If you prefer to run commands manually:

```bash
# Start services in background
docker-compose up -d

# View logs
docker-compose logs -f mariadb
docker-compose logs -f php

# Stop services
docker-compose down

# Stop and remove volumes (WARNING: deletes all data)
docker-compose down -v
```

## Production Deployment (cPanel)

### 1. Create Database in cPanel

1. Log into cPanel → MySQL Databases
2. Create a new database (e.g., `parentdataforce_db`)
3. Create a user with full privileges
4. Note the credentials (database name, username, password)

### 2. Import Schema

**Option A: Using phpMyAdmin**
1. Navigate to phpMyAdmin in cPanel
2. Select your database
3. Click "Import"
4. Upload `backend/schema.sql`
5. Upload `backend/admin_schema.sql`

**Option B: Using command line (SSH)**
```bash
mysql -u username -p database_name < backend/schema.sql
mysql -u username -p database_name < backend/admin_schema.sql
```

### 3. Seed Initial Data

Generate the seed SQL from your current case metadata:
```bash
python backend/export_metadata_sql.py
```

Then import:
```bash
mysql -u username -p database_name < backend/seed_from_metadata.sql
```

### 4. Create Admin User

You need to create at least one admin user. Run this SQL (replace `your_password_hash` with a hash generated by PHP's `password_hash()`):

```sql
INSERT INTO admin_users (username, email, password_hash, full_name, role, status)
VALUES ('admin', 'admin@yourdomain.com', 'your_password_hash', 'Administrator', 'owner', 'active');
```

To generate a password hash:
```php
<?php echo password_hash('your_secure_password', PASSWORD_DEFAULT); ?>
```

### 5. Configure PHP Admin

1. Upload the `admin/` directory to your hosting (e.g., `public_html/admin`)
2. Update `admin/includes/config.php` with your database credentials
3. Ensure `.env` file is not publicly accessible (outside web root)
4. Configure `.htaccess` for security (see `admin/.htaccess.example`)

## Schema Overview

### Core Tables
- `districts` - School districts
- `cases` - Public records requests and investigations
- `case_events` - Timeline events for cases
- `documents` - Linked documents/files
- `case_links` - Relationships between cases
- `ingest_message_ids` - Email ingestion deduplication
- `ingest_file_hashes` - File deduplication

### Admin & Security Tables
- `admin_users` - User accounts with roles
- `admin_sessions` - Active login sessions
- `admin_audit_log` - Audit trail of all admin actions
- `csrf_tokens` - CSRF protection tokens

## Maintenance

### Updating Schema
When schema changes are needed:

1. Create a migration SQL file in `backend/migrations/`
2. Update the main schema files if this is a new deployment
3. For existing databases, run the migration manually

### Backing Up Data
```bash
# Using mysqldump
mysqldump -u username -p database_name > backup_$(date +%Y%m%d).sql

# Using Docker
docker exec pdf_mariadb mysqldump -u pdf_user -p pdf_db > backup.sql
```

### Restoring Data
```bash
mysql -u username -p database_name < backup.sql
```

## Troubleshooting

### Database Connection Issues
- Verify MariaDB is running: `docker-compose ps`
- Check logs: `docker-compose logs mariadb`
- Test connection: `python backend/test_database.py`
- Ensure `.env` credentials match docker-compose.yml

### PHP Admin Issues
- Check PHP error log: `docker-compose logs php`
- Verify PHP extensions: `docker exec pdf_php php -m | grep pdo`
- Ensure Apache is serving the correct directory

### phpMyAdmin Issues
- Ensure MariaDB is healthy before phpMyAdmin starts
- Check connection settings in phpMyAdmin config

## Security Notes

### For Development
- Default passwords are weak - change them!
- `.env` file contains credentials - keep it out of version control
- The admin interface is only accessible locally (ports 8080/8081)

### For Production
- Use strong, unique passwords for database users
- Store database credentials outside the web root
- Enable HTTPS for all admin access
- Regularly update passwords and review audit logs
- Implement IP whitelisting for admin access if possible
- Keep PHP and MariaDB updated
